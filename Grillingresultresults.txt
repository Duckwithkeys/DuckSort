PROLOGUE
--------
I went through the entire Views/, Views/Components/, and Utilities/ tree.
Some parts are competent (the thumbnail loader is genuinely well thought-out,
and the routing preview is a nice idea). But the visual and interaction layer
is held together with hardcoded numbers, magic paddings, two competing design
languages, and Apple HIG violations stacked on top of each other. This is
not a "ship it" product. This is a senior dev's first SwiftUI project that
was never put in front of a designer. Let's get into it.

================================================================================
1. FIRST IMPRESSIONS — WHAT LOOKS AMATEUR RIGHT AWAY
================================================================================

1.1  YOU HAVE TWO DESIGN SYSTEMS AND BOTH ARE LEAKING.
     - `PhotomatorTheme` in LiquidGlassStyle.swift uses a proper
       adaptive NSColor(name: nil) { appearance in ... } pattern with named
       tokens (background, sidebarBackground, cellBackground, etc.). Good.
     - Then `SettingsPaneWindow.swift` invents `dsColor("#1E1E1E")`,
       `dsColor("#323232")`, `dsColor("#3C3C3E")` — hardcoded hex strings
       scattered through every settings pane. There is also a `Color(hex:)`
       extension somewhere used by `CustomTag.swift`.
     - The footer uses `Color(red: 0.12, green: 0.12, blue: 0.12)` in
       FilmstripView (line 71) — a *third* hardcoded gray.
     - The toolbar background token exists but is unused. The big viewer
       background is `PhotomatorTheme.background` while the filmstrip is
       a near-black that doesn't match anything in the theme.
     Result: the main app is "Photomator dark", the filmstrip is "darker
     than Photomator", the settings window is "yet another dark". This is
     a fragmentation bug, not a styling choice.

1.2  ADAPTIVE THEME IS HALF-BAKED.
     You went to the trouble of making PhotomatorTheme adaptive (line 15–87),
     but then the grid's "JPEG" badge is `Color.white` text on `Color.white`
     background. In light mode that is invisible. See PhotoSetCell.swift:43–48.

1.3  INCONSISTENT CORNER RADIUS.
     12pt on the grid thumbnail (PhotoSetCell.swift:24). 8pt on the empty
     state cards (EmptyLibraryView.swift:40). 6pt on the search field
     (SidebarView.swift:82). 4pt on the filmstrip cell (FilmstripView.swift:89).
     There is no design system. There's just "whatever radius I typed today".

1.4  FONT SIZES ARE POINTILLISM.
     The "tag hotkey" text is `.system(size: 10, weight: .bold)` in one place,
     `.caption2.weight(.medium)` in another, `.caption` in a third. The
     "ACTIVE TAGS" / "LIBRARY" / "TAGS" all-caps captions mix `.caption.weight(.bold)`
     with `.caption2` and `.system(size: 10, weight: .bold)`. There is no
     typographic scale. You made one up as you went.

1.5  TRAFFIC-LIGHT SPACING IS HARD-CODED IN THREE PLACES.
     - SidebarView.swift:14 → `Spacer().frame(height: 38)`
     - LargeImageViewer.swift:76 → `Spacer().frame(width: 72)`
     - ContentView.swift:37 → `.ignoresSafeArea(.container, edges: .top)`
     None of these agree. If Apple changes the title bar height in macOS 16,
     three files break independently.

1.6  THE BRAND BAR IS AWKWARD.
     SidebarView.swift:14–47: 38pt of dead space, then a 24×24 logo, then
     the app name. The icon image-search fallback chain (lines 18–34) is
     16 lines of defensive code for a static asset that should be one
     `Image("duck_logo")`. And `Bundle.module` is for SwiftPM resources,
     not for app bundle resources — for an `.app` use `Bundle.main`. This
     will silently fail in production and the fallback chain hides the
     failure.

1.7  THE "PREMIUM LINEAR GRADIENT SEPARATOR".
     SidebarView.swift:50–57. You literally call it a "premium" gradient
     in the comment. It is a 1pt tall gradient from selected blue to
     blue-at-10% alpha. It is not premium. It is a hairline that is also
     wider than the search field's separator (0.5 visual weight), so the
     eye reads the search field as floating in space, then meets a
     decorative line. Hierarchy is broken before the user does anything.

2. SWIFTUI IMPLEMENTATION FLAWS
================================================================================

2.1  GeometryReader TO GET A WIDTH IS A CODE SMELL.
     PhotoGridView.swift:34–112 wraps the entire scroll view in a
     `GeometryReader { geometry in ... }` solely to compute a column count
     and stash it in the view model. Problems:
     - GeometryReader is the heaviest layout primitive SwiftUI has. You
       are paying for it on every redraw.
     - `viewModel.gridColumnCount` is then *consumed* by ContentView's
       key handler to compute arrow-key wrap-around (ContentView.swift:204).
       The whole arrow-key math is coupled to a value derived from the
       initial layout, but it can change mid-session.
     - The right primitive is `LazyVGrid`'s built-in adaptive columns. If
       you really need a count for keyboard math, compute it from
       `geometry.size.width` once, store it, and `coordinateSpace` is your
       friend. Better: do the arrow math with a wrapping function that
       recomputes columns lazily from a `@State` width.

2.2  HARDCODED PADDINGS EVERYWHERE.
     ContentView.swift:448 → `.padding(12)` on the empty state.
     ContentView.swift:455 → TransferFooter with no horizontal padding of
     its own beyond `.padding(.horizontal, 16)` inside.
     PhotoGridView.swift:47–55 → `.padding(.top, 44)` for the grid header.
     PhotoGridView.swift:87 → `.padding(.horizontal, 20)`.
     These numbers (12, 16, 20, 44) are not on a 4pt or 8pt grid. 44pt
     for the top is "approximately the title bar height" — that's what
     safeAreaInsets is for. Use them. Or use a constant.

2.3  THE GRID HEADER REINVENTS `safeAreaInset(edge: .top)`.
     PhotoGridView.swift:37–55 manually pads 44pt at the top, draws a
     one-pixel `Rectangle().fill(PhotomatorTheme.selectedBlue)` to act
     as a divider, then 16pt more padding. This is exactly what
     `.safeAreaInset(edge: .top) { ... }` is for, and it would also
     auto-respect the window's title bar — which currently your code
     *assumes* is 44pt. (It isn't, on every Mac, and not in fullscreen.)

2.4  YOU'RE USING A `Rectangle()` AS A DIVIDER.
     PhotoGridView.swift:51–55, TransferFooter.swift:115–120,
     LargeImageViewer.swift:181–185. SwiftUI has `Divider()`. Use it.
     The custom one is fine when you need a specific color, but you
     don't — the theme's `separator` token exists for exactly this.

2.5  BAD STACK NESTING.
     PhotoSetCell.swift:18–122 nests an outer `VStack` around the
     thumbnail, the text labels, the tag pills, and a `Spacer(minLength: 0)`
     *plus* `.frame(maxHeight: .infinity, alignment: .top)`. The frame
     modifier is fighting the Spacer, and inside a `LazyVGrid` cell you
     don't want either. Let the cell be intrinsic; the grid will give it
     height.
     - The result of this is that tag pills get clipped in narrow columns
       and there's no way to know without printing. You wrote
       `Spacer(minLength: 0)` to "fix" the symptom, but the disease is
       the maxHeight: infinity.

2.6  THE BORDER MODIFIER REINVENTS A STANDARD PRIMITIVE.
     LargeImageViewerSidebar.swift:196–223 ships 28 lines of `EdgeBorder`
     to draw a one-pixel leading border. SwiftUI has `Divider()` for
     vertical dividers in an HStack, or just `Rectangle().frame(width:1)`
     between the two columns in ContentView. 28 lines of geometry to
     avoid `Rectangle().frame(width: 1)`.

2.7  CUSTOM BUTTON STYLES THAT REINVENT `ButtonStyle(.borderless)`.
     OutlineButtonStyle in SettingsPaneWindow.swift:229–248 re-implements
     a bordered button. SwiftUI has `.buttonStyle(.bordered)`. Your
     `AccentCapsuleButtonStyle` is `.buttonStyle(.borderedProminent)` with
     a tint. Stop.

2.8  `NSApp.keyWindow?.makeFirstResponder(nil)` IS EVERYWHERE.
     PhotoGridView.swift:68, 75, 82, 92. SidebarView.swift:99, 124, 201,
     315, 561. EmptyLibraryView.swift:68. This is your "deselect the
     text field" hack and it appears 9 times. The fact that you need
     to call it from tap handlers means your focus model is wrong.
     SwiftUI's `FocusState` (which you *are* using for search, line 10)
     can be reset at the source. The NSEvent local monitor in
     `ContentView.registerKeyboardMonitor` should be the single place
     that decides "if a text field is first responder, do nothing".
     Right now your handlers fire anyway and the only reason it works
     is the `isFirstResponderTextField(in:)` check at the top of
     `handleGlobalKeyPress`. So you have BOTH the early-return AND the
     `makeFirstResponder(nil)` calls. Pick one. The right one is
     "early-return".

2.9  THE GESTURE STACK ON THE GRID CELL IS A MESS.
     PhotoGridView.swift:72–84 stacks two `.simultaneousGesture` calls
     (single tap, double tap) and a Button in `PhotoSetCell` (line 19).
     Single tap → toggleSelection. Double tap → openLargeImageViewer.
     Single tap *also* fires on every double tap on macOS, because
     `simultaneousGesture` doesn't suppress the inner button's action.
     Result: double-clicking a photo opens the viewer *and* toggles
     selection. The user has just "selected" an image they were trying
     to preview. I would bet real money this is a real bug in the
     product right now. Use `Button` with a long-press for double-tap,
     or use `NSGestureRecognizer` semantics via `.onTapGesture(count: 2)`
     with a real single-tap fallback only on the first event in a window.

2.10 `LazyVGrid` + `.id(photoSet.id)` FOR SCROLL TARGETS IS FINE, BUT...
     PhotoGridView.swift:95–104 dispatches the scroll-to work via
     `DispatchQueue.main.async` inside `onChange`. The `withAnimation`
     wraps a `scrollProxy.scrollTo` which is a side-effecting layout
     operation, not a state change. Animation here will *fight* the
     scroll. Use `Task { try? await Task.sleep(...); scrollTo(...) }`
     or just call `scrollTo` directly. You're overcomplicating it.

2.11 THE FORMATTING "ITEM / ITEMS" STRING.
     PhotoGridView.swift:43 → `"\(viewModel.filteredPhotoSets.count)
     item\(count == 1 ? "" : "s")"`. This is `String.localizedStringWithFormat`
     territory, or at minimum use `Text("^[\(count) item](inflect: true)"`
     in iOS 16+/macOS 13+. You're shipping English-only pluralization.

2.12 THE INSPECTOR HAS A BROKEN STACK.
     LargeImageViewer.swift:29–50 wraps the inspector in a `ZStack`
     *inside* the image pane, with `.padding(12)` on the inspector
     and `.background(Color.black).clipShape(RoundedRectangle(12))`
     on the same outer container three times (lines 32, 41, 51, 52).
     The `.background(Color.black).clipShape(...)` is repeated on the
     "no photos" placeholder, on the inner ZStack, and on the outer
     VStack. You are clipping the same rect three times. One is enough.

2.13 `.frame(maxHeight: .infinity, alignment: .top)` ON A BUTTON'S LABEL.
     PhotoSetCell.swift:123. The button is in a `LazyVGrid` cell. The
     grid will give every cell equal height. You're forcing the cell
     to be as tall as possible. Result: in wide windows you get
     comically large gaps below the text. The "Spacer(minLength: 0)"
     on line 120 then exists only to fill that gap. Both can be deleted.

2.14 THE LARGEST OFFENDER: `PhotomatorTheme` NAMES LIE.
     The theme is named "Photomator" but this app is "DuckSort". This
     will not pass any review and is the first thing a senior iOS
     reviewer notices — naming is part of design system discipline.
     Rename to `Theme` or `DuckSortTheme`.

3. UX & INTERACTION NIGHTMARES
================================================================================

3.1  THE SORTING INTERFACE HAS NO "SORT" BUTTON. IT HAS A "TRANSFER" FOOTER.
     TransferFooter.swift:62–83: Copy / Move / Symlink / Hardlink actions
     sit alongside a "Choose Destination..." button. To the user, this
     is the action bar. To the model, this is "the transfer bar that
     happens to do routing". The labeling "Choose Destination..." is
     a verb-phrase setup but the actual primary action ("Copy", "Move",
     etc.) is hidden behind a system image. The user has to learn four
     icons. New users will not know what "tray.and.arrow.down" with
     a folder icon means. Use text labels. Always.

3.2  THE GRID HEADER SAYS "X ITEMS" BUT THE FILTER RULE NAME IS A STRING.
     PhotoGridView.swift:39: `viewModel.filterRule.rawValue`. So the
     header reads "All Photos" / "34 items". The user does not know
     what "items" means. Is it photos? Photo sets? Files? The footer
     correctly says "X photo sets selected, Y files" (TransferFooter.swift:21).
     Pick one term. The grid is rendering `PhotoSet`s, so call them
     "sets" or "photos" — not "items".

3.3  GRID CELL VISUAL HIERARCHY IS A PILE OF BADGES.
     PhotoSetCell.swift:40–98 stacks FIVE overlays on the thumbnail:
     - Top-left: format badge (white pill, "RAW")
     - Top-left: wand icon (orange) OR warning icon (red)
     - Bottom-left: rating capsule
     - Bottom-right: pick flag
     - Plus a hover border, a selection border, a focus border.
     In a 180pt wide cell on a typical MacBook, this is a 6-element
     pin-cushion. The eye doesn't know where to look first. The format
     badge competes with the rating for "primary metadata". Either
     collapse the format + status into a single chip, or move rating
     to below the cell with the title (which is where users expect it
     in a grid).

3.4  REJECTED PHOTOS BECOME GRAY AND FADED.
     PhotoSetCell.swift:25–26: `grayscale(0.8); opacity(0.6)` for
     pick == -1. Then the focus border still draws at full opacity,
     and the format badge stays bright white. So a rejected photo
     in focus is "faded ghost with a glowing blue ring and a
     screaming white RAW badge". That contradicts itself.

3.5  THE THUMBNAIL HOVER STATE IS 0.15 ALPHA WHITE.
     PhotoSetCell.swift:30: `isHovered ? Color.white.opacity(0.15) : Color.clear`.
     On a 180pt cell in a dark window, this is *barely visible* on
     a dark photo and *invisible* on a light photo. Pick a border
     color that contrasts with the photo, or use a 2pt white inner
     glow on the bottom edge — Photomator does exactly this for a
     reason. Or use `.shadow` with a small radius.

3.6  THE SEARCH FIELD IS 12PT FONT ON 11PT ICON.
     SidebarView.swift:60–88. 12pt text in a 24pt tall pill. This is
     smaller than Apple's macOS sidebar search field (which is 13pt)
     and smaller than the body text in the grid. Hierarchy is inverted:
     the most-used control in the sidebar is also the smallest text.

3.7  THE SIDEBAR HAS NO "ACTIVE FILTER" INDICATOR BEYOND ROW COLOR.
     When the user has 3 ratings + 2 flags + 2 tag categories active,
     there's no global "you have 5 filters active, click to clear"
     affordance. They have to scan the sidebar, remember what they
     clicked, and untoggle. A toolbar "Active Filters (5) · Clear"
     row at the top would save dozens of confused users.

3.8  EMPTY STATE IS SENT TO `EmptyLibraryView` *AND* HAS ITS OWN SCAN STATE.
     ContentView.swift:444–449 only shows the empty state when
     `viewModel.photoSets.isEmpty`. But the scanning progress lives
     inside EmptyLibraryView (line 14) with its own `isScanning` prop.
     If the user adds a source and the scan starts, the empty state
     *goes away* (because photoSets is no longer empty once the first
     file lands) and the scanning message disappears mid-scan. There
     is no transition. This is a "where did my progress bar go" bug.

3.9  INSPECTOR "I" KEY AND LARGE VIEWER "I" KEY BOTH TOGGLE INSPECTOR.
     ContentView.swift:155–158 (viewer) and 256–259 (grid) both bind
     `i` to `viewModel.isInspectorOpen.toggle()`. That's fine. But
     in the viewer, the inspector slides in from the *leading* edge
     INSIDE the image pane (LargeImageViewer.swift:46–48), while in
     the grid there's no inspector concept. So the user opens the
     viewer, hits I, and the photo's right half is suddenly covered
     by a 240pt wide panel that is *also* sitting on top of the
     image. The right sidebar (LargeImageViewerSidebar) is 260pt
     wide and to the right. So the user has TWO sidebars fighting
     for screen space. This is bad. Pick one inspector surface.

3.10 LARGE IMAGE VIEWER ZOOM/INSPECT IS UNDOCUMENTED.
     There is no visible UI hint that you can do anything in the
     viewer besides arrow around. Where is the rating control? Where
     is the pick flag? Where is the export preview? It is all in the
     right sidebar — but the right sidebar's section headers are
     "ACTIVE TAGS", "IMAGE METADATA", "ROUTED EXPORT PREVIEW" with
     no visual indication of *what action* the user is supposed to
     take. This is an inventory, not a UI.

3.11 THE FILMSTRIP IS A LIGHTER VERSION OF THE GRID, BUT IT INVENTS A
     NEW THUMBNAIL SIZE (120×80 frame to 72×48 render).
     FilmstripView.swift:87. The `ThumbnailView` is asked for
     `size: CGSize(width: 120, height: 80)` but then `.frame(72, 48)`
     is applied. The internal loader does `kCGImageSourceThumbnailMaxPixelSize`
     based on the *requested* size (120) not the *displayed* size (72).
     Result: you decode a 2.7× larger image than you need. On a 5K
     iMac with a 600-cell library, this is hundreds of MB of waste
     memory in your cache (ThumbnailCache.swift:153 → 80MB cap).
     Use the actual display size.

3.12 KEYBOARD FOCUS MODEL IS INVISIBLE.
     `viewModel.focusedPhotoIndex` is mutated by arrow keys, by
     single-clicks, by the filmstrip, and by the viewer's left/right
     arrows. There is no persistent visual "this is the focused
     photo" affordance outside the grid. When the user opens the
     large viewer, the blue focus ring on the grid is hidden. When
     they close the viewer, the focus is still on the same photo —
     good — but the scroll-to on open (PhotoGridView.swift:95–104)
     fires *again*, which can scroll the grid out from under them.

3.13 THE TRANSFER FOOTER STACKS RULE STATUS UNDER STATUS MESSAGE.
     TransferFooter.swift:15–55: status text on top, "X photo sets
     selected" subtitle, destination button. Then BELOW that, rule
     summary. Then BELOW that, action buttons. Then BELOW that,
     progress. This is four vertical sections fighting for attention.
     Apple's own Mail / Photos apps keep the primary action (Send /
     Transfer) anchored to the trailing edge, with secondary status
     inline. You have the primary action (Copy/Move/Symlink) floating
     in the middle, and the destination button next to the status.
     Reverse the visual order: action buttons right, rule preview
     left, status in the middle, progress at the very bottom.

3.14 SHORT-CUT POPOVER USES UPPERCASE BOLD FOR SECTION HEADERS.
     ContentView.swift:326 ("APP ACTIONS"), 359 ("CULLING CONTROL"),
     377 ("TAG HOTKEYS"). 10pt bold, all caps, secondary color.
     Apple HIG: section headers in a list should use the system
     "Section" style, not custom all-caps text. On macOS, system
     settings panes use 11pt regular or 13pt semibold, not 10pt bold.
     You're shouting in a 340pt wide popover.

3.15 THE DUCK LOGO IS RESIZED 24×24 INSIDE A 24×24 FRAME.
     SidebarView.swift:36–38: `.frame(width: 24, height: 24)`. The
     logo is a raster PNG being scaled to fit. If it's an SF Symbol
     style icon, you should use the system symbol API. If it's a
     brand mark, it should be vector or 3x retina. As written, it
     will look soft on Retina displays with `aspectRatio(contentMode: .fit)`
     — and `aspectRatio` on an `Image` is fine, but with no `.interpolation(.high)`
     it's bilinear by default. Add `.interpolation(.high)`.

4. APPLE HUMAN INTERFACE GUIDELINES VIOLATIONS
================================================================================

4.1  macOS APP WINDOW WITHOUT A PROPER TOOLBAR.
     ContentView.swift uses a custom title bar via `ignoresSafeArea(.container, edges: .top)`.
     You are then manually inserting 38pt (sidebar) and 44pt (grid)
     of "title bar space". The HIG says: use `.toolbar` with
     `ToolbarItem` placements, set `.navigationTitle`, and let AppKit
     draw the unified title bar. You are currently hand-rolling the
     entire window chrome, which means:
     - Window controls may overlap your content on non-standard title bars
     - The window's title is empty (`navigationTitle("")` on line 15)
     - The window cannot be styled by the user (translucent, hidden)
     - Fullscreen transition will not be smooth

4.2  `Color.accentColor` MIXED WITH HARDCODED BLUE.
     LargeImageViewer.swift:152 uses `Color.accentColor` for the info
     button. FilmstripView.swift:93 uses `Color.accentColor` for the
     focused filmstrip border. But everywhere else uses
     `PhotomatorTheme.selectedBlue = Color(red: 0.251, green: 0.537,
     blue: 1.0)` — a specific shade. If the user changes the system
     accent color, the info button changes but the focus ring does
     not. The user expects the focus ring and the system accent to
     match. They will not. Pick one. (Pick `Color.accentColor`.)

4.3  "0" KEY TO CLEAR TAGS — UNDOCUMENTED macOS CONVENTION VIOLATION.
     ContentView.swift:262–274: pressing `0` clears all tags, rating,
     and pick on the focused photo. In Adobe Lightroom, the "0" key
     is the standard "no rating" key, and this is fine. But the user
     has *no way to discover this*. There's no menu item, no tooltip,
     no on-screen affordance. The popover in `ShortcutsPopoverView`
     lists it (line 365) — good — but the popover is hidden behind
     a "?" button that I don't even see referenced in the main UI.
     Add a Help menu item and a toolbar help button.

4.4  CUSTOM FOCUS RING INSTEAD OF `FocusRingView` / `.focused()`.
     PhotoSetCell.swift:35–38: `if isFocusedGridItem` draws a 2.5pt
     blue border. macOS has `NSView.focusRingType` and SwiftUI has
     `.focused()` (and on macOS, `@FocusState` for collections).
     Your custom ring is offset 2.5pt from a 3pt selection border
     (line 31) — when both are active, you get a 5.5pt thick border
     around a 180pt cell. That's *half* the cell width as border.
     Use `.focusEffectDisabled(false)` (macOS 14+) or at least
     reduce the focus ring to 1.5pt and only show it when the
     selection border is also present.

4.5  `textFieldStyle(.plain)` ON THE SEARCH FIELD.
     SidebarView.swift:64–65. macOS text fields have a default
     rounded-rectangle style that matches the system. You replaced
     it with a custom 6pt-radius pill, then re-drew the border with
     `PhotomatorTheme.separator`. This is fine for a custom design,
     BUT you set the placeholder color to nothing (you don't even
     set `.foregroundStyle` on the placeholder) and the clear button
     appears mid-text instead of at the trailing edge, because you
     have a `HStack` with no trailing alignment logic. This violates
     the "search field" UI pattern users know from Finder, Mail,
     Safari, Photos.

4.6  NO HOVER FEEDBACK ON THE GRID CELL ITSELF.
     PhotoSetCell.swift:124–129 updates `isHovered` but the only
     consumer is the white border on line 30. There's no scale,
     no shadow, no lift. Compare to Photos.app, which on macOS
     gently scales the thumbnail to 1.03× and adds a subtle drop
     shadow on hover. Your cell goes from "nothing" to "barely
     visible white hairline". Users will think their cursor isn't
     over the cell.

4.7  KEYBOARD SHORTCUTS CONFLICT WITH SYSTEM SHORTCUTS.
     ContentView.swift:147 binds `S` to "toggle selection". This
     collides with `Cmd+S` (Save) and `S` while a text field is
     focused. Your `isPlainKey` check (line 147) prevents it from
     firing with modifiers, but it will still fire on a bare `S`
     press inside the search field if the focus check fails. And
     the `S` shortcut is also a tag shortcut for the "S" tag in
     TagManager. So you have THREE things mapped to `S`: select
     toggle, tag apply, and any tag whose key is "S". A user with
     a tag named "S" cannot use the toggle-select shortcut. There
     is no warning at tag-creation time. This is a foot-gun.

4.8  TAB KEY FOR CATEGORY CYCLING.
     ContentView.swift:136–139 (viewer) and ContentView.swift:206
     (grid) — wait, the grid doesn't have Tab mapped. The viewer
     does. Tab is the system "next focusable element" key on macOS.
     Intercepting Tab to cycle categories will break VoiceOver
     focus traversal and any user's muscle memory. This is the
     single biggest HIG violation in the file. Use `\` or `[`/`]`
     for category cycling, or use a button with `keyboardShortcut`.

4.9  `Cmd+A` INTERCEPTED BEFORE THE SYSTEM HANDLER.
     ContentView.swift:103–109: you intercept `Cmd+A` and call
     `viewModel.selectVisiblePhotoSets()`. On macOS, `Cmd+A` is
     "Select All" for the current text field. If the user is in
     the search field and presses `Cmd+A`, your handler will
     select all photos instead of selecting the search text. The
     `isFirstResponderTextField` early-return (line 96) saves you,
     but only because you put it there. This is exactly the
     pattern that causes the "I lost my text" bug reports.

4.10 THE TRANSFER FOOTER'S "RULE:" LABEL USES A LITERAL "•".
     TransferFooter.swift:134. `Text("•")`. This is a literal
     Unicode bullet as a UI separator. HIG: use `Divider()` or
     a system symbol that scales with Dynamic Type. Also, the
     rule path can be very long and you only `.lineLimit(1)` it
     (line 142) — the user will see "Rule: Wedding · /Volumes/
     Pho…" and have no way to read the full path. TruncationMode
     is `.tail` by default. Use `.middle` for paths.

4.11 SETTINGS WINDOW IS NON-RESIZABLE.
     SettingsPaneWindow.swift:75 → `.frame(minWidth: 720, minHeight: 480)`.
     The comment at line 6 calls this "Safari-style preferences"
     — Safari's preferences *are* resizable. The "non-resizable"
     comment is contradictory and the window is also too small
     for the Tags pane (SettingsTagsPaneView, 15KB of code that
     probably doesn't fit in 480pt). HIG: preference windows
     should be resizable with a sensible minimum.

4.12 THE TAGS PANE IMPORTS CONTACTS FROM A "FOOTER" BUTTON.
     SettingsPaneWindow.swift:66–73, 137–207. A "Import Contacts…"
     button in the footer of the Tags pane opens a vCard picker
     and bulk-creates tags. There is no undo. There is no
     confirmation. There is no "merge with existing" vs "replace"
     choice. If a user accidentally imports the wrong file, they
     will get 500 new "John Smith" tags. The HIG says: destructive
     or bulk-modifying actions need confirmation.

================================================================================
SUMMARY
================================================================================
What you have: a working photo culling app with a competent thumbnail
loader, a real keyboard-driven workflow, and a useful export-routing
preview. The engineering underneath is solid.

What you don't have: a design system, focus discipline, HIG compliance,
or a clear visual hierarchy. The current UI looks like a developer
mimicked Photomator's colors and Lightroom's shortcuts without
understanding either.

Time-to-fix if you focus on the high-priority items: ~1 week for
sections 5.1–5.10 and 5.13–5.15. Ship those and the app will look
and feel like a real Mac app.

Don't ship the rest of section 5 first. Section 5.1 is the load-bearing
wall. Everything else is decoration on top of it.
