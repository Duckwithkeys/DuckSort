DUCKSORT BACK-END ARCHITECTURAL TEARDOWN & SECURITY AUDIT

Prepared by: Senior Back-End Architect & Security Engineering

1. PERFORMANCE & LATENCY
Synchronous EXIF Extraction & Heavy Processing:
If your backend processes images synchronously upon upload, your worker nodes will choke immediately. EXIF extraction via naive libraries often reads the entire file into memory instead of streaming header blocks. For high-resolution RAW and 4K images, decoding pixels on the primary API thread will block event loops and exhaust CPU limits. You must offload thumbnail generation and EXIF reading to asynchronous serverless workers or dedicated background job queues (like SQS with auto-scaling worker pools).

Thumbnail Generation Overhead:
Generating multiple thumbnail resolutions on-the-fly without an aggressive edge CDN caching layer will destroy your compute budget. Image resizing is CPU-bound; doing it repeatedly per request is architectural suicide. Thumbnails must be generated once asynchronously, stored in object storage with immutable content-hash keys, and served strictly via a global CDN.


2. DATABASE & INDEXING
Scaling to Millions of Photos:
Storing photo metadata in a monolithic relational table without partitioning will cause query degradation at scale. Queries filtering across combinations of date ranges, geographic bounding boxes, and arbitrary user tags will result in full table scans unless multi-column composite indexes and spatial indexing (such as PostGIS or QuadTrees) are configured properly.

Pagination Flaws:
Offset-based pagination (OFFSET 100000 LIMIT 50) is fundamentally broken for large datasets. As offset increases, database engines must scan and discard thousands of rows, leading to linear latency spikes. You must enforce cursor-based pagination (keyset pagination) using indexed columns like creation timestamp and unique photo ID.


3. STORAGE & I/O
Upload Pipeline Bottlenecks:
Routing file uploads directly through your application API servers wastes bandwidth, memory, and application connections. API servers should never proxy heavy binary data. Use presigned S3/GCS URLs for direct client-to-bucket uploads, combined with multipart upload protocols for large media files.

Cloud Storage Costs:
Keeping full-resolution uncompressed photos in hot storage tiers indefinitely will bankrupt the operation. Implement lifecycle management policies to move infrequently accessed originals to cold storage (e.g., S3 Glacier Flexible / Instant Retrieval) while serving everyday previews from warm storage tiers and edge caches.


4. SECURITY & PRIVACY
Insecure Direct Object References (IDOR):
If photo URLs or API endpoints rely on sequential integer IDs or predictable hash schemes without validating object-level ownership, unauthorized users can iterate through keys and access private user photos. Every request for image data or metadata must enforce strict tenant isolation and cryptographic authorization checks at the database/middleware layer.

Data Leakage via EXIF Metadata:
Exposing raw EXIF metadata directly to public or shared links leaks highly sensitive PII, including exact GPS coordinates, camera serial numbers, and home location patterns. EXIF metadata must be sanitized or stripped upon public sharing, keeping private metadata accessible strictly to the authenticated asset owner.

Insecure Presigned URLs:
Generating presigned URLs with long expiration windows allows leaked URLs to be accessed indefinitely by unauthorized third parties. Presigned URLs for viewing or downloading must have tight expiration windows (e.g., 5 to 15 minutes) and be bound to restricted IP ranges or user sessions where feasible.


5. ALGORITHMIC EFFICIENCY
Clustering and Duplicate Detection:
Performing perceptual hashing or visual feature extraction in real-time inside request-response cycles is completely unviable. Spatial clustering algorithms like DBSCAN or K-Means for location grouping must not run on raw coordinates during database reads; pre-aggregate geohashes or spatial clusters offline in batch pipelines.

Naive Photo Sorting:
Sorting large libraries by complex multi-variable rankings (e.g., visual similarity, face detection clusters, and chronological metadata) at query time will melt database nodes. All sorting indexes, tag associations, and cluster memberships must be pre-calculated and stored in dedicated search indices (like OpenSearch or Elasticsearch) rather than dynamically joined across relational tables on every user request.
